## 🔍 使用 Git 发现和分析 PR（核心方法）

**仓库已克隆**：你当前的工作目录就是仓库根目录，可以直接使用 git 命令发现和分析所有 PR。

### ⭐ 步骤1: 发现已 merged 的 PR

**⚠️ 重要说明**：以下命令只查找已经合并到当前分支的 PR（merge commits）

**方法A: 查找主分支的 merge commits** (推荐，确保是已 merged 的)
```bash
# 确保在主分支上查找（main/master）
git checkout main 2>/dev/null || git checkout master 2>/dev/null

# 查找所有 merge commits (这些都是已经 merged 的 PR)
git log --merges --oneline --max-count=50

# 查找包含 "Merge pull request" 的 commits (GitHub 标准格式)
git log --grep="Merge pull request" --oneline --max-count=50
```

**方法B: 查找最近的 merge commits 详细信息**
```bash
# 获取最近30-50个 merge commits 的完整信息
git log --merges --format="%H|%s|%an|%ae|%ad" --max-count=50

# 格式说明：
# %H = commit SHA (这就是 merge_commit_sha)
# %s = subject (标题)
# %an = author name
# %ae = author email
# %ad = author date
```

### ⭐ 步骤2: 分析每个 PR 的代码改动

对于提供的PR列表（或你用git log发现的PR），使用以下命令深度分析：

1. **查看 PR 的代码改动**：
   ```bash
   # 查看完整的代码 diff
   git diff <base_commit>..<merge_commit>

   # 查看改动的文件列表
   git diff --name-only <base_commit>..<merge_commit>

   # 查看统计信息（新增/删除行数）
   git diff --stat <base_commit>..<merge_commit>

   # 查看改动的 Verilog/SystemVerilog 文件
   git diff --name-only <base_commit>..<merge_commit> | grep -E '\.(v|sv|svh|vh)$'
   ```

2. **分析具体文件**：
   ```bash
   # 查看所有 Verilog/SV 文件的改动
   git diff <base_commit>..<merge_commit> -- '*.v' '*.sv' '*.svh' '*.vh'

   # 查看测试文件的改动
   git diff <base_commit>..<merge_commit> -- '*_tb.v' '*_tb.sv' 'tb_*' '*test*'

   # 查看 cocotb 测试的改动
   git diff <base_commit>..<merge_commit> -- 'test_*.py' '*cocotb*'
   ```

3. **检测新增文件**（重要！）：
   ```bash
   # 列出新增的文件
   git diff --diff-filter=A --name-only <base_commit>..<merge_commit>

   # 统计新增的 Verilog/SV 文件
   git diff --diff-filter=A --name-only <base_commit>..<merge_commit> | grep -E '\.(v|sv|svh|vh)$' | wc -l
   ```

4. **检查仿真工具依赖**（CRITICAL！）：
   ```bash
   # 检查是否依赖商业 EDA 工具
   git diff <base_commit>..<merge_commit> | grep -iE 'vcs|xrun|irun|vsim|questa|ncvlog|xmsim|dc_shell'

   # 检查 Makefile 中的仿真器设置
   git diff <base_commit>..<merge_commit> -- 'Makefile' '*.mk' | grep -iE 'SIMULATOR|SIM\s*='

   # 检查是否使用开源工具
   git diff <base_commit>..<merge_commit> | grep -iE 'iverilog|verilator|cocotb|vunit|ghdl'
   ```

### ⭐ 步骤3: 智能分析策略

**如果PR列表太少**（比如只提供了5-10个PR）：
1. 使用 `git log --merges` 发现更多 PR
2. 解析 merge commit 标题提取 PR 编号
3. 对每个 PR 使用 git diff 分析代码变更
4. 筛选出最适合的 New Feature 类型 PR

**建议**：优先分析最近的30-50个 merge commits，这样既能覆盖足够的PR，又不会太慢。

# New Feature 任务类型定义（Verilog/SystemVerilog 特定）

### 1. 新增 RTL 模块
**特征**：
- 添加新的 Verilog/SystemVerilog 模块（module）
- 新增完整的数字逻辑功能单元

**典型改动**：
- 新增 `.v` / `.sv` 源文件定义新模块

- 实现组合逻辑或时序逻辑
- 有对应的 testbench 文件

**典型文件模式**：
- 新增 `*.v` / `*.sv` 设计文件
- 对应的 `*_tb.v` / `*_tb.sv` testbench 文件

- 实现片上互联（NoC, crossbar）

**典型改动**：
- 新增接口模块（master, slave, arbiter, decoder）
- 协议状态机实现
- 握手信号处理（valid/ready, req/ack）

### 3. 新增功能单元
**特征**：
- 添加 FIFO、Buffer、Cache 等存储结构
- 新增 ALU、乘法器、除法器等运算单元
- 添加 DMA、中断控制器等系统组件
- 实现编解码器、CRC、加密模块等

**典型改动**：
- 新增功能模块文件
- 参数化设计（parameter, localparam）
- 有对应的验证环境

### 4. 功能增强和扩展
**特征**：
- 在现有模块基础上添加新端口或新功能
- 扩展参数化配置选项
- 添加新的操作模式或状态
- 支持新的数据宽度或格式

**典型改动**：
- 为模块添加新的 parameter
- 扩展状态机的状态
- 添加新的接口端口
- 增加新的控制寄存器

### 5. 新增验证组件
**特征**：
- 添加可复用的验证 IP（VIP）

- 新增 testbench 基础设施

- 创建验证辅助模块

# Bugfix 任务类型定义（Verilog/SystemVerilog 特定）

**仅当 task_type = "bugfix" 或 "all" 时适用**

## ✅ 明确属于 Bugfix 类型的 PR

### 1. RTL 逻辑修复
**特征**：
- 修复组合逻辑或时序逻辑中的错误
- 修正状态机的状态转换错误
- 修复信号赋值或位宽不匹配问题
- 修正边界条件处理（如计数器溢出、FIFO 满/空判断）

**典型改动**：
- 修改 `always_ff`、`always_comb`、`assign` 语句中的逻辑
- 修正 `if/else`、`case` 条件分支
- 修复信号位宽截断或符号扩展错误
- 有对应的 regression testbench 验证修复

**典型 PR 标题模式**：
- `fix:`, `bug:`, `hotfix:`, `bugfix:`
- "Fix FSM deadlock", "Correct FIFO overflow handling"
- "Fix AXI handshake violation", "Resolve timing issue"

### 2. 协议合规性修复
**特征**：
- 修复总线协议违规（AXI/Wishbone/APB 握手错误）
- 修正时序违规（setup/hold violation, 多周期路径）
- 修复跨时钟域同步问题
- 修正 reset 行为不符合规范

**典型改动**：
- 修正 valid/ready 握手逻辑
- 添加缺失的同步器（synchronizer）
- 修复 reset 后的初始状态
- 修正协议信号的时序关系

### 3. 边界条件和极端情况修复
**特征**：
- 修复空/满条件处理
- 修正溢出/下溢检测
- 修复仲裁器（arbiter）饥饿或死锁问题
- 修正数据路径的 corner case

**典型改动**：
- 添加缺失的边界检查
- 修正 off-by-one 错误
- 修复 race condition

### 4. 功能性 Bug 修复
**特征**：
- 修复模块输出错误的数据
- 修正计算结果不正确
- 修复控制信号行为异常
- 修正中断处理错误

**典型改动**：
- 修改数据路径逻辑
- 修正控制信号生成
- 修复寄存器读写逻辑

## ❌ 不属于 Bugfix 类型的 PR

- **新增功能**：添加原本不存在的模块或接口（这是 New Feature）
- **重构**：仅重组代码但不修复任何 bug
- **性能优化**：仅改善时序/面积，原始逻辑没有错误
- **文档更新**：仅更新注释或 README
- **代码风格**：仅格式化、信号重命名

## Bugfix PR 的特殊筛选标准

1. **必须有可复现的 bug**：PR 描述中应包含 bug 的症状或错误现象
2. **必须有对应的测试**：PR 应包含 regression test（testbench 或 cocotb 测试）验证 bug 被修复
3. **修复应是代码级的**：不能只是配置或约束的修改
4. **改动范围适中**：bug 修复通常改动较小（10-500 行），过大的改动可能不是纯 bugfix
5. **base commit 有 bug → 测试失败；target commit 修复了 → 测试通过**

**如果 task_type = "bugfix"**:
- 检查是否真的修复了一个 RTL bug（不是重构或优化）
- 分析 bug 的清晰度：是否能从 PR 描述中理解 bug 是什么
- 评估修复是否有对应的 regression test
- 评估 bug 的复现性：base_commit 是否能稳定复现 bug

**如果 task_type = "all"**:
- 识别 PR 属于 new_feature 还是 bugfix
- 按对应类型的标准评分

### 2. 测试覆盖评分 (0-10)
- 是否有新增 testbench 文件？（new_feature）或 regression test？（bugfix）
- 仓库是否有 cocotb 测试覆盖？
- testbench 是否能验证新功能？
- 是否可以用开源工具运行测试？

### 3. 任务质量评分 (0-10)
- PR 是否聚焦单一新功能？
- 新功能是否容易理解和描述？
- 是否能提取出清晰的用户需求？

### 4. 仿真复杂度评分 (0-10)
- **简单仿真** (+2分)：标准 iverilog，无特殊依赖
- **中等复杂** (0分)：需要 verilator 或 cocotb 配置
- **复杂仿真** (-2分)：需要特殊 IP 核或复杂环境
- ❌ **需要商业 EDA**：扣5分，通常直接排除

## 第三轮：排序和推荐

综合评分 = 任务类型相关性 * 0.4 + 测试覆盖 * 0.3 + 任务质量 * 0.2 + 仿真复杂度 * 0.1

**🔴 最终推荐前必须二次确认**：
- 在推荐 PR 之前，**必须再次检查该 PR 是否依赖商业 EDA 工具**
- 使用 `git diff <base>..<merge>` 仔细查看代码变更
- 检查 Makefile 和脚本中的仿真器设置
- 如果发现任何商业 EDA 依赖（即使是少量），**立即排除该 PR**
- 宁可不推荐任何 PR（返回空列表），也不要推荐依赖商业 EDA 的 PR

选出评分最高的 **Top {top_n}** 个 PR（必须是开源工具可运行）

# 特殊说明

## 评分标准
- overall_score >= 8.5: 优质 PR，强烈推荐
- overall_score >= 7.0: 合格 PR，可以使用
- overall_score < 7.0: 不推荐，不加入候选列表

## Verilog/SystemVerilog 特定注意事项

1. **仿真工具检测**：
   - Icarus Verilog: `iverilog -g2012`, `vvp` 命令
   - Verilator: `verilator --cc --exe --build`
   - cocotb: `import cocotb`, `SIM=icarus` in Makefile

2. **测试框架检测**：
   - cocotb: `test_*.py` 文件 + `import cocotb`
   - UVM (开源部分): `import uvm_pkg`
   - VUnit: `from vunit import VUnit`
   - 原生 testbench: `*_tb.v`, `*_tb.sv`

3. **新模块文件**：
   - 新增 `.v` / `.sv` 文件通常意味着新的 RTL 模块
   - 这是 New Feature 的强烈信号

4. **参数化设计**：
   - 带 `parameter` 的模块更适合作为任务（有丰富的配置选项）
   - 参数化设计通常测试性更好

5. **接口类型**：
   - 标准总线接口（AXI, Wishbone, APB）的实现适合作为任务
   - 自定义接口需要有清晰的文档或规范
