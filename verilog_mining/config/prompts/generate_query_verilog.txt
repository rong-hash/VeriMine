基于给定的两个 commits 之间的代码变更，生成一个自然的用户查询（user query），用于 New Feature 类型的硬件设计编程任务。

## 核心挑战

**你可以看到 target_commit**，这给你带来：
- ✅ **优点**：准确理解任务范围、设计需求、接口规格
- ❌ **缺点**：容易不自觉地把实现细节写进 query

**你的任务**：
1. 充分利用 target_commit 来**理解设计需求**
2. 然后**假装忘记**实现细节，用**用户视角**描述需求
3. 生成的 query 应该像"需要新功能的硬件工程师"写的，而非"看过 RTL 代码的人"写的

## 任务类型定义

**必须匹配指定的任务类型**: {task_type}

- **New Feature**: 设计原本没有的新 RTL 模块、接口、协议实现或功能单元
- **Bugfix**: 修复已有 RTL 模块中的逻辑错误、时序问题、协议违规或功能缺陷

**根据任务类型调整 query 风格**:

如果 task_type = "new_feature":
- query 描述需要**什么新功能**，像工程师提出新需求
- 重点：模块接口规范、功能描述、参数化要求

如果 task_type = "bugfix":
- query 描述遇到了**什么 bug/问题**，像工程师报告 bug
- 重点：错误现象描述、复现步骤、预期 vs 实际行为
- **不要说如何修复**，只描述问题！
- 示例："AXI slave 模块在 burst 写入结束后 BVALID 信号没有拉高，导致 master 端一直等待写响应"

## 生成要求

### 语言风格
- Use natural Chinese with English technical terms (Verilog keywords, protocol names)
- Be direct and task-oriented but comprehensive
- Match real hardware engineer patterns: describe module interfaces, signal behaviors, timing requirements
- **NO POLITE GREETINGS**: Do NOT start with "你好", "Hi", "Hello"
- **Start directly with the task** - real engineers jump straight to the design requirement

### 1. RL 可评估性 (CRITICAL)

**生成的 query 必须能够生成明确的 rubrics（评判标准）**：

✅ **可评估的 query**：
- "实现一个 AXI4-Lite slave 接口模块，支持 32 位数据宽度，地址空间 0x0000-0x00FF，包含 4 个可读写寄存器"
- "添加一个参数化的同步 FIFO 模块，支持可配置深度和数据宽度，包含 full/empty/almost_full 标志"

❌ **不可评估的 query**：
- "优化总线接口" / "改进 FIFO 设计" / "重构模块结构"

**判断标准**：
- 是否能列出 3-10 个明确的完成标准？
- 每个标准是否可以通过仿真/testbench 验证？
- 提供足够技术细节让任务可理解、可执行、可评估

### 2. 用户视角 (CRITICAL)

**你是提出需求的硬件工程师，不是看过 RTL 代码的人！**

硬件工程师的特点：
- 工程师**不知道**要改哪些 Verilog 文件
- 工程师**不知道**具体的内部信号名和状态机编码
- 工程师只知道：需要什么功能 / 模块接口规格 / 时序要求

### 3. 禁止泄露"未来信息" (CRITICAL)

#### ✅ 用户可能知道的信息（可以包含）：

**New Feature 和 Bugfix 通用**：
- **模块接口规范**：端口名称、信号宽度、方向（input/output）
- **协议规范**：AXI4 规范要求的信号、时序关系
- **时序约束**：如 "单时钟域"、"异步复位"、"输出寄存一拍"
- **预期行为**：如 "读空 FIFO 时 underflow 标志应该拉高"

**New Feature 特有**：
- **功能需求**：需要实现什么功能（如 "支持 burst 传输"、"实现写入缓冲"）
- **参数化要求**：如 "数据宽度可配置"、"FIFO 深度参数化"

**Bugfix 特有**：
- **错误现象**：如 "仿真时 FIFO 在满状态下仍然接受写入，导致数据丢失"
- **复现条件**：如 "连续写入 16 次后，第 17 次写入的数据覆盖了第 1 个位置"
- **预期 vs 实际**：如 "预期 full 信号在 16 个数据后拉高，实际没有"
- **错误信息**：如 "testbench 报告 assertion failed: data_out !== expected_data"
- **影响范围**：如 "这个问题只在 burst 长度 > 8 时出现"

#### ❌ 只有看过 target_commit 才知道的信息（绝对禁止）：
- **具体的 RTL 修复方案**：❌ "将 `>=` 改为 `>`"（Bugfix）
- **具体的 RTL 实现**：❌ "使用 gray code 跨时钟域"（New Feature）
- **具体的信号名**：❌ "内部信号 `wr_ptr_gray` 用于..."
- **具体的状态编码**：❌ "状态机使用 3 位编码，IDLE=3'b000"
- **具体代码行**：❌ "在 always_ff 块的第 42 行添加..."
- **具体的 bug 根因**：❌ "问题是因为 counter 溢出没有处理"（只说现象，不说根因！）
- **新增的内部模块**：❌ target_commit 中新创建的内部子模块名称

**判断标准**：问自己"一个没看过 RTL 代码的硬件工程师，会这么说吗？"

### 4. 硬件设计特定的信息披露

**✅ 应该包含的硬件设计信息**:
- 模块名称和顶层端口列表（这是接口规范，不是实现细节）
- 协议遵从性要求（如 "符合 AXI4 规范"）
- 参数化选项（parameter 名称和含义）
- 时钟域要求（单时钟、多时钟、异步）
- 复位策略（同步/异步、高/低有效）
- 新增文件的路径（如 "在 rtl/ 目录下新建 fifo.sv"）

**❌ 不应包含的信息**:
- 具体的组合逻辑表达式
- 内部状态机的状态编码
- 内部流水线级数
- 具体的 always 块实现

### 5. 自然表达 (CRITICAL)

**禁止写成技术规格文档或数据手册！**

❌ **严禁使用这些写法**:
- **文档化格式**: markdown 表格列出所有端口、小节式规格书
- **啰嗦铺垫**: "我在看了下代码"、"根据需求分析..."
- **过度列举**: 列举所有信号、所有状态、所有边界条件
- **设计判断**: "我觉得应该用 gray code"
- **反问句**: "能不能..."、"可以吗..."

✅ **应该像真实硬件工程师一样**:
- **单刀直入**: 直接说需要什么模块和功能
- **点到为止**: 给出关键接口规格，不过度展开
- **需求导向**: 说"需要一个 FIFO 模块"，而不是"设计一个 FIFO 的 RTL"
- **长度控制**: 150-500字

### 6. 其他要求

- **避免商业 EDA**: 不涉及 VCS、Questa、Xcelium 等商业工具
- **需求导向**: 描述需要什么功能，不描述如何实现
- **信息完整**: 提供足够的接口规格让工程师理解任务
- **完整覆盖** (CRITICAL): 确保 query 覆盖所有核心 RTL 变更

## 生成流程

**New Feature (task_type = "new_feature")**:
1. **理解变更**：查看 `git diff`，分析新增了什么模块、接口、功能
2. **推断需求**：从"开发者实现了什么 RTL"反推"硬件工程师需要什么功能"
3. **用户视角**：假装没看过 target_commit，工程师只知道需求和接口规格
4. **自然表达**：用流畅的段落描述，不要写成规格文档

**Bugfix (task_type = "bugfix")**:
1. **理解修复**：查看 `git diff`，分析修复了什么 bug，涉及哪些模块
2. **推断问题**：从"开发者修复了什么"反推"用户遇到了什么问题"
3. **用户视角**：假装不知道 bug 的根因和修复方案，只描述观察到的问题现象
4. **自然表达**：像工程师在报告 bug，描述现象、复现条件、预期行为
5. **不要说如何修复**：只描述"什么坏了"和"应该怎样才对"

**通用自检**：
- 信息来源检查：每句话工程师是怎么知道的？（接口规格/错误现象 ✅ vs 内部实现/修复方案 ❌）
- 可评估性检查：能列出 3-10 个明确的完成标准？
- 表达方式检查：是像在沟通需求/报告 bug 还是像在写数据手册？
- 问自己："没看过 RTL 代码的硬件工程师，会这么说吗？"
