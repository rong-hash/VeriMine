## 核心改进原则

### 1. 🚨 接口规范提示 (CRITICAL - 避免假阴性)

**这是最重要的改进点！**

**假阴性问题**：
- Agent 实现了等价的 RTL 功能但使用了不同的模块名
- Testbench 因找不到指定模块名或端口名而编译失败
- 功能正确但 testbench 无法实例化模块
- **Agent 创建了新 RTL 文件但放在了错误的路径**

**分析流程**：

```bash
# 步骤 1: 查看 testbench 代码，识别实例化的模块和连接的端口
git diff {base_commit}..{target_commit} -- "*_tb.v" "*_tb.sv" "tb_*" "test_*.py"

# 步骤 2: 检查这些模块/端口在 base_commit 中是否存在
git checkout {base_commit}
grep -r "module 模块名" . --include="*.v" --include="*.sv"
```

**处理规则**：

| 接口 | base_commit 中存在 | 处理方式 |
|------|:---:|---------|
| `existing_module` | ✅ 存在 | 不需要在 query 中说明 |
| `new_module` | ❌ 不存在 | **必须**在 query 中提供接口规范 |

**接口规范应包含**：
- 模块名称（module name）
- 关键端口名称和方向（input/output, 信号宽度）
- 参数名称和用途（parameter）
- 模块应放置的文件路径

**🚨 新增 RTL 文件路径 (CRITICAL)**：
如果原始变更新增了设计文件（不是 testbench），**必须**在改进后的 query 中明确：
- 新文件的完整路径（如 `rtl/axi_fifo.sv`）
- 新文件包含的模块名（如 `module axi_fifo`）

```
❌ 缺少新文件路径（会导致 Agent 放错位置或不创建文件）：
"添加一个 AXI FIFO 模块"

✅ 明确新文件路径：
"添加一个 AXI FIFO 模块。
新建文件 rtl/axi_fifo.sv，包含 module axi_fifo，
端口包括 AXI4-Stream slave 输入接口和 AXI4-Stream master 输出接口。"
```

**改进示例**：

```
❌ 原始 query（缺少接口规范，会导致假阴性）：
"需要为项目添加一个数据缓冲模块。"

✅ 改进后（添加接口规范，不暴露 testbench 细节）：
"需要为项目添加一个数据缓冲模块。

接口要求：
- 新建文件 rtl/data_buffer.sv，包含 module data_buffer
- 参数：DATA_WIDTH（数据位宽，默认32）, DEPTH（缓冲深度，默认16）
- 输入端口：clk, rst_n, wr_en, wr_data[DATA_WIDTH-1:0]
- 输出端口：rd_data[DATA_WIDTH-1:0], full, empty
- 时钟域：单时钟域，同步复位"
```

### 2. 用户视角转换 (CRITICAL)

**你看到了 target_commit 的 RTL 代码，但硬件工程师没有！**

硬件工程师只知道：
- ✅ 需要什么功能的模块
- ✅ 模块的接口规格（端口、参数）
- ✅ 时序和协议要求
- ✅ 仿真中观察到的错误现象

硬件工程师不知道：
- ❌ 具体的 RTL 实现（always 块、组合逻辑）
- ❌ 内部信号命名
- ❌ 状态机编码方式

**检查清单 - 保留 vs 移除**:
- [保留] 新模块的名称和端口列表（当 testbench 依赖时）
- [保留] 新 RTL 文件应该放置的路径
- [保留] 参数名称和含义
- [移除] testbench 文件路径和 cocotb 测试函数名
- [移除] 具体的 RTL 实现（如"用 gray code"、"用 BRAM"）
- [移除] 内部信号名和状态编码

**转换方法**:
```
❌ 错误（泄露 RTL 实现）:
"在 rtl/fifo.sv 的 always_ff 块中，
使用 gray code 编码的读写指针进行跨时钟域同步，
通过双级触发器链消除亚稳态..."

✅ 正确（保留接口规范，隐藏实现细节）:
"需要一个异步 FIFO 模块，支持不同的读写时钟域。
模块名 async_fifo，放在 rtl/async_fifo.sv。
参数包括 DATA_WIDTH 和 ADDR_WIDTH。
端口需要分别的读写时钟和复位信号。"
```
